# Chat LMS Agent Self-Maintaining Harness

## TL;DR
> **Summary**: Replace the current string-matched CLI and oversized bootstrap script with a small deterministic harness: command modules, a durable tool registry, scoped memory, hook-driven hydration/closeout, and isolated private profile state. The agent remains the interpreter; the harness stores, validates, lists, executes safe reusable tools, and reinjects concise inventory into future sessions.
> **Deliverables**:
> - A real `chat-lms` command tree for tools, memory, hooks, context, doctor, session, and bootstrap generation.
> - A durable registry for tool drafts, active tools, memory entries, and decision records.
> - SessionStart hydration that tells Codex which tools exist and how to use them.
> - Stop/PostCompact closeout that verifies tool/memory changes were persisted.
> - Tests and PowerShell CLI-capture QA evidence for every behavior.
> **Effort**: Large
> **Parallel**: YES - 5 waves
> **Critical Path**: Task 1 -> Task 2 -> Task 4 -> Task 5 -> Task 7 -> Final Verification

## Context
### Original Request
The harness must let the agent create, manage, remember, and reuse CLI tools across sessions. A new Codex session must automatically know the available tool inventory. The current harness does not do this.

### Current Failure
- `src/chat_lms_agent/cli.py` only hand-matches a few argument lists and cannot scale to a tool collection.
- `hooks/hooks.json` contains a Stop command for `session closeout`, but that command is not implemented.
- `src/chat_lms_agent/context.py` returns static placeholder context and does not hydrate tool inventory or durable memory.
- `scripts/bootstrap.ps1` owns profile generation, hook generation, context rendering, and self-sync logic directly, instead of delegating to reusable CLI commands.
- Existing tests mostly assert key/file/string presence; they do not execute every hook command or validate CLI contracts.

### Reference Patterns Incorporated
- Memory and tools are separate concepts: memory stores stable facts/preferences; tools store reusable procedures.
- Reusable tools follow a lifecycle: draft -> validate -> activate -> list -> deprecate.
- Durable state should be searchable and compactly reinjected into future sessions.
- The harness should stay small and inspectable. It is not a model router, chat platform, or natural-language parser.
- Hook systems should call stable commands, not embed large procedural scripts.
- Stop hooks may block completion when durable work remains incomplete.

### Defaults Applied
- Parser: Python stdlib `argparse`, no runtime dependency added in the first implementation.
- Storage: public repo uses fixtures and temp state only; real runtime state lives under private profile paths generated by bootstrap.
- Tool kinds for v1: `query_template`, `action_template`, `workflow`, `module_command`.
- Safe self-creation for v1: the agent may draft and activate data-driven tools (`query_template`, `action_template`, `workflow`). Arbitrary new Python code generation creates a draft and test plan only, then requires explicit implementation review.
- Platform: Windows-first because current bootstrap is PowerShell. Cross-platform shell bootstrap is later work.
- Delete is out of scope. Use `deprecate` to retire tools.

## Work Objectives
### Core Objective
Build a deterministic, privacy-safe harness that makes reusable CLI tool inventory and durable operational memory available across Codex sessions.

### Deliverables
- Command router and subcommand modules.
- Runtime profile path resolver and private state store.
- Tool registry models, validation, lifecycle, and snapshot output.
- Memory store, scoped retrieval, compaction, and closeout verification.
- Hook command package for SessionStart, UserPromptSubmit, PostToolUse, PostCompact, and Stop.
- `profile inspect` command for safe runtime path inspection.
- `scripts/qa/capture-command.ps1` wrapper that captures command, exit code, stdout, stderr, and cleanup notes.
- Thin PowerShell bootstrap that delegates generation to Python CLI.
- Contract tests, privacy tests, temp-profile integration tests, and CLI-capture QA transcripts.

### Definition of Done
- `uv run pytest -q` exits 0.
- `uv run ruff check .` exits 0.
- `uv run basedpyright` exits 0.
- `uv run python -m chat_lms_agent --help` lists the command tree.
- `uv run python -m chat_lms_agent tool list --json --profile test-fixture` exits 0 and returns JSON.
- `uv run python -m chat_lms_agent context hydrate --for-codex --json --profile test-fixture` includes active tools and scoped memories with secrets redacted.
- `uv run python -m chat_lms_agent session closeout --verify-memory --json --profile test-fixture` exits 0 when registry/memory obligations are satisfied and nonzero when they are not.
- Every command referenced by `hooks/hooks.json` is executed by an automated test.
- Bootstrap integration tests generate a private temp workspace without writing to real profile locations.

### Must Have
- Structured JSON output for all agent-facing commands.
- Deterministic exit codes:
  - `0`: success
  - `2`: validation or contract error
  - `3`: approval required
  - `4`: unsafe or privacy boundary violation
  - `5`: closeout blocked
- Atomic state writes and simple lock files for registry/memory writes.
- Redaction tests for credentials, private paths, and generated memory text.
- No private runtime data committed to the public repo.

### Must NOT Have
- No natural-language command grammar.
- No giant script that implements runtime behavior directly.
- No custom daemon, model router, messaging platform, or tool zoo.
- No arbitrary Python code generation and activation without explicit implementation/test review.
- No real external writes in tests or dry-runs.
- No destructive local changes without explicit approval.

## Verification Strategy
> ZERO HUMAN INTERVENTION - all verification is agent-executed.
- Test decision: TDD. Each task starts with failing tests that name the contract.
- QA policy: each task includes CLI-surface QA evidence. Tests alone do not complete a task.
- Evidence directory: `evidence/self-maintaining-harness/`
- Privacy policy: evidence must use temp profiles and sample fixtures only.
- CLI QA standard: every CLI-shaped QA scenario uses `powershell -ExecutionPolicy Bypass -File scripts/qa/capture-command.ps1 -Name <scenario> -Command '<literal command>' -Evidence <path>`. The wrapper must record command, working directory, exit code, stdout, stderr, and cleanup receipt. This replaces tmux for Windows workspaces where tmux is unavailable.
